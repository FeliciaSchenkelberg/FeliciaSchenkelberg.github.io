Binary to C++

In this explanation, I will be converting a simple example of binary code to assembly code. Then again translating this assembly code to C++ code. So, the question begins? What happens if we are given an unopenable, binary file and told to translate this legacy code to something a little more modern (like C++)? Well, the first step is to collect as much information as you can about the software (i.e., source design documents).

What if no such documentation exists? Well, that means we need to put on our white hats and reverse engineer this program! It is important to first translate this code to assembly. Simple enough, right?

Bash (Unix shell)

cd /home/codio/workspace/Project1Files
ls
objdump -S Project1.o > Project1.s

Now our program is openable! This is because we have translated it into assembly language. A lower-level programming language. If we could translate our binary code to assembly so simply, why then couldn’t we use a simar command to translate our binary code to C++? Great question! We can’t yet translate binary to such a high-level language. While assembly language is a higher-level language than binary, it is still a lower-level language than higher level programming languages like C#, Visual Basic, C, C++, JavaScript, Objective C, BASIC and Pascal. These languages are more developed than low-level languages and closer to a human spoken language.

Now that we have a general idea of the functionality, it is important to record data flow and identify where security vulnerabilities may be present. Common security vulnerabilities include but are not limited to:

  • Stack Overflow
  • String Filter
  • Integer Overflows
  • Type conversion errors

Stack Overflow is when security threats take advantage of the fact that programmers often neglect to control the bounds of incoming data. A simple stack overflow vulnerability could occur when a program receives data from the outside world and naively copies that data into the stack without controlling its’ length.

A significant percentage of overflow attacks have also been associated with strings. The most common example is the use of various runtime library string-manipulation routines for copying or processing strings in some way while letting the routine determine how much data should be written.

Integer overflows a type of overflow error where incorrect treatment of integers can lead to a numerical overflow, ultimately leading to a buffer overflow. The most common example of this is when an application receives the length of some data block from the outside world.

The semantics of a programming language can be critical because they determine how programs handle data at a lower level. Type conversion errors occur when developers mishandle incoming data types performing incorrect conversions.

One way to determine the appropriate fix for a security vulnerability is to use a threat modeling system, such as STRIDE. This acronym stands for Spoofing, Tampering, Repudiation, Information Disclosure, Denial of Service, and Elevation of Privilege. To determine the appropriate fix, you must first decompose the system into relevant components. This is done to be able to analyze the threat sensitivity of each component and mitigate it.
