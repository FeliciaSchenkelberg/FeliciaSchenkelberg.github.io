Software Reverse Engineering

Software Reverse Engineering is a process of recovering the design, requirement specifications and functions of a product from an analysis of its code. Security researchers reverse-engineer code to find security risks in programs. They also use the technique to understand malicious applications and disrupt them. But researchers aren't the only ones doing this: bad actors also want to find software flaws through reverse engineering.

Prior to this lesson, it was my understanding that security vulnerabilities were faults in information systems, programs, and applications that could be exploited by threat sources. I have since learned that security means that the system has the properties of confidentiality, integrity, and availability. Additionally, users must be authenticated and authorized correctly. This leads me to the conclusion that security vulnerabilities are a deficiency of one or more of these properties: confidentiality, integrity, or availability which can allow threat sources to exploit the given information system, program, or application.

As mentioned above, vulnerabilities are essentially faults in information systems, programs, or applications that compromise the security of a program. This can allow threat sources to exploit the system. In most cases, the kinds of vulnerabilities identifiable in C++ code are external. Meaning that they come from the outside world. This includes input such as files downloaded to the program or even data packets sent over a network. The idea is to provide unexpected program inputs to get the program to stray from its’ normal execution path. Two such examples of vulnerabilities include but are not limited to integer overflows and type conversion errors.

I should be looking for vulnerabilities during legacy to C++ conversion rather than during testing, as it will be easier for me to secure the program. During testing, there are no ways to test for every possible code path for bugs and security holes (Microsoft, 2019). By looking for vulnerabilities during legacy to C++ conversion, I can easily express the architecture and design requirements of the application, maintain code optimization and efficiency, as well as ensure code security. In this way, I eliminate many faults on which many threat sources rely.

One way to determine the appropriate fix for a security vulnerability is to use a threat modeling system, such as STRIDE. This acronym stands for “Spoofing, Tampering, Repudiation, Information Disclosure, Denial of Service, and Elevation of Privilege” (Microsoft, 2019). To determine the appropriate fix, you must first decompose the system into relevant components. This is done to be able to analyze the threat sensitivity of each component and mitigate it.
